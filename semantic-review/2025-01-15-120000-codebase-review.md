# Production Readiness Review: Daymenify Digital Product Platform

Daymenify is a digital product marketplace (pulsa, game top-ups, e-wallets, PLN tokens) built on Express + Prisma + BullMQ + Redis + Socket.io. It integrates multiple payment gateways (Tripay, Midtrans, Xendit, Duitku, BayarGG, Pakasir) and multiple providers (Digiflazz, VIP-Reseller, Tokovoucher) through adapter patterns with circuit-breaking and smart routing. The architecture is sound in its abstractions, but several gaps in race condition handling, credential management, webhook security, schema mismatches, and missing production infrastructure make it unsuitable for production deployment without remediation.

**Watch for:** Race condition between payment webhook and order processing allows double-spend on the same transaction (confirmed). Provider credentials stored in plaintext in the database (confirmed). `redis.keys('products:*')` in sync worker will block Redis in production (confirmed). Auth service schema assumes `firstName`/`lastName` fields that don't exist on the Prisma User model (confirmed). No webhook IP allowlisting or rate limiting on webhook endpoints (confirmed).

## High-level view

The payment and provider adapter layer is well-structured with a clean interface contract, timing-safe signature verification, and encrypted config storage for payment gateways. However, provider credentials (`apiKey`, `apiSecret`) are stored as plaintext columns on the `Provider` model while payment gateway configs get AES-256-GCM encryption — an inconsistency that leaves provider secrets exposed to any database read.

The order processing pipeline has a critical race condition: the webhook handler updates transaction status to PAID and immediately queues an order job, but the order worker also accepts transactions in PAID status. If the webhook fires twice before the idempotency key is set (network retry within milliseconds), or if the order worker picks up the job before the status update commits, duplicate provider orders can be submitted against a single payment.

The sync worker uses `redis.keys('products:*')` which scans the entire keyspace — a blocking O(N) operation that will stall all Redis operations under load. The markup recalculation in the same worker updates products one-by-one in a loop rather than using batch transactions, making it both slow and non-atomic.

The auth system implements good patterns (bcrypt, refresh token rotation, httpOnly cookies, timing-safe comparison for token reuse detection) but has a schema mismatch: the service writes `firstName`/`lastName` while the Prisma schema only has a single `name` field. This will crash at runtime. The RBAC middleware checks `role` as a string on the JWT but the database schema uses a `roleId` foreign key to a Role table — the JWT role claim is never validated against the actual permission tree.

Socket.io is configured without a Redis adapter, meaning realtime events won't work across multiple server instances. The code comments acknowledge this but it's commented out, making horizontal scaling of the WebSocket layer impossible without code changes.

1. **Race condition in payment→order pipeline** — Transaction status check in order worker (`status !== 'PROCESSING' && status !== 'PAID'`) combined with webhook setting status to PAID and queuing the job creates a window for duplicate orders. Add a Redis lock or database advisory lock before submitting to provider.
2. **Provider credentials in plaintext** — `Provider.apiKey` and `Provider.apiSecret` are stored unencrypted in PostgreSQL. Apply the same AES-256-GCM encryption used for `PaymentGateway.config`.
3. **`redis.keys()` in sync worker** — Replace `redis.keys('products:*')` with `SCAN` or use a Redis Set to track cache keys, avoiding O(N) blocking.
4. **Auth schema mismatch** — `auth.service.ts` writes `firstName`/`lastName` but Prisma schema has only `name`. This will throw at runtime. Align the schema or the service.
5. **No webhook rate limiting or IP allowlisting** — Webhook routes are unprotected by rate limiters. Attackers can flood the audit_log table or attempt signature brute-force. Add rate limiting and restrict source IPs.
6. **Webhook handler uses `global` for factory access** — `(global as any).__paymentFactory` is brittle and breaks under test. Use dependency injection or a module-scoped singleton.
7. **Markup recalculation is non-atomic** — Individual `prisma.product.update` calls in a loop mean partial failures leave prices inconsistent. Use `prisma.$transaction` for batch updates.
8. **Socket.io without Redis adapter** — Realtime events won't propagate across multiple Node instances. Enable the Redis adapter before horizontal scaling.
9. **No transaction-level locking on status transitions** — `prisma.transaction.update` without a WHERE on current status allows concurrent webhooks to overwrite each other. Use optimistic locking (`UPDATE ... WHERE status = 'PENDING'`).
10. **RBAC doesn't check actual permissions** — JWT `role` string is compared directly; the `RolePermission` table in the schema is never queried. Either simplify the schema or implement permission lookups.
11. **Env validation missing provider/payment secrets** — `env.ts` only validates generic vars. Provider API keys, encryption keys, and gateway callback URLs are not validated at startup, causing runtime crashes.
12. **Graceful shutdown doesn't drain BullMQ workers** — Workers are started but never closed in the shutdown handler, risking in-flight job corruption.
13. **Notification worker channels are placeholders** — Email, Telegram, and Push handlers log placeholders but jobs will succeed silently, misleading monitoring.
14. **Cron sync job uses `providerId: 'all'`** — The sync worker expects a real provider ID and will fail to find the provider record, silently skipping syncs.

## Race conditions in the payment-to-order pipeline

The payment webhook handler (`payment.webhook.ts:77-85`) performs the idempotency check via `redis.set(key, '1', 'EX', 86400, 'NX')` and then updates the transaction and queues the order job. The order worker (`order.worker.ts:48-51`) guards on status being `PROCESSING` or `PAID`. This creates two race windows:

First, between the Redis NX set and the Prisma update committing: if the same webhook arrives on two server instances within a few milliseconds, both may pass the Redis check before either writes. This is unlikely but possible under Redis cluster partitioning.

Second, and more critically, the order worker doesn't use optimistic locking. It reads the transaction, checks status, and then updates — but doesn't use a WHERE clause on the current status in the update. If a provider webhook fires (completing the order) while the order worker is mid-processing, both the worker and the provider webhook handler will attempt conflicting status transitions.

```typescript
// order.worker.ts:60 - no optimistic lock
await prisma.transaction.update({
  where: { id: transactionId },
  data: { status: 'PROCESSING' },
});
```

Recommended fix: Use `prisma.transaction.update({ where: { id, status: 'PAID' }, data: { status: 'PROCESSING' } })` and check the returned count. If zero rows affected, another process already moved the state forward.

## Provider credential storage asymmetry

The `PaymentGatewayFactory` (`services/payment/index.ts:76-107`) implements proper AES-256-GCM encryption for gateway configs. The encrypted string format (`iv:authTag:ciphertext`) and the decryption flow are correctly implemented.

However, `Provider` model fields (`prisma/schema.prisma:146-149`) store `apiKey`, `apiSecret`, and `webhookSecret` as plain strings:

```prisma
model Provider {
  apiUrl        String
  apiKey        String      // ← plaintext
  apiSecret     String?     // ← plaintext
  webhookSecret String?     // ← plaintext
}
```

The `ProviderFactory.buildConfig` (`services/provider/index.ts:72-80`) reads these directly without decryption. Any database access (backup, SQL injection, admin panel leak) exposes all provider API credentials.

## Blocking Redis operation in sync worker

`sync.worker.ts:148-151`:
```typescript
const cacheKeys = await redis.keys('products:*');
if (cacheKeys.length > 0) {
  await redis.del(...cacheKeys);
}
```

`KEYS` is O(N) where N is the total number of keys in Redis, not just matching ones. With thousands of cached entries, this blocks the Redis event loop for all clients. Replace with `SCAN` iteration or maintain a Redis Set of product cache keys for O(1) lookup and deletion.

## Auth service vs Prisma schema mismatch

`auth.service.ts:80-84` creates a user with:
```typescript
const user = await prisma.user.create({
  data: {
    email: email.toLowerCase(),
    password: hashedPassword,
    firstName,  // ← doesn't exist in schema
    lastName,   // ← doesn't exist in schema
    role: 'USER', // ← schema uses roleId FK
  },
});
```

The Prisma schema (`schema.prisma:17-37`) defines:
```prisma
model User {
  name     String       // single field, not firstName/lastName
  roleId   String       // FK to Role, not a string enum
  role     Role @relation(...)
}
```

This code will crash with a Prisma validation error on any registration attempt. The schema also requires `referralCode` (non-nullable, unique) which the auth service never provides.

## Webhook endpoints lack access controls

The webhook router (`webhooks/index.ts`) registers routes directly without any middleware. While the handlers verify signatures internally, there's no:
- Rate limiting (the `defaultRateLimiter` on the main app would apply, but at 100 req/15min this is too generous for webhooks)
- IP allowlisting (payment gateways publish their callback IPs)
- Request body size limits specific to webhooks

Each webhook creates an `auditLog` entry before signature verification (`payment.webhook.ts:38-45`). An attacker can flood the audit_log table with garbage by sending requests with invalid signatures to any webhook endpoint.

## Global factory pattern fragility

Both webhook handlers access adapters via `(global as any).__paymentFactory` and `(global as any).__providerFactory`. This pattern:
- Has no type safety
- Breaks in test environments where globals aren't initialized
- Creates hidden coupling between the app bootstrap and webhook handlers
- Returns `null` silently when the factory isn't set, swallowing real initialization failures

The payment webhook handler responds 200 even when the adapter is null (`payment.webhook.ts:49-53`), which means if the factory initialization fails at startup, all payment webhooks will silently succeed without processing.

## Markup recalculation scalability

`markup/index.ts:186-218` loads ALL active products into memory, then iterates and updates them one by one:

```typescript
const products = await prisma.product.findMany({
  where: { status: 'ACTIVE' },
  include: { providerProducts: { ... } },
});

// Then for each product:
await prisma.product.update({ where: { id }, data: { sellingPrice } });
```

With thousands of products, this creates thousands of individual UPDATE queries. The batch boundary (`BATCH_SIZE = 100`) only controls iteration, not the actual database operations. Each individual update can fail, leaving the catalog in a partially-updated state. This should use `prisma.$transaction` per batch.

## Graceful shutdown incompleteness

`index.ts:30-48` handles SIGTERM/SIGINT by closing the HTTP server, disconnecting Prisma, and disconnecting Redis. However:
- BullMQ workers (started via `startWorkers()`) are never closed, risking in-flight job corruption
- Socket.io connections are not explicitly closed
- The 30-second force timeout doesn't account for long-running provider API calls in the order worker

## Notification channels as silent no-ops

`notification.worker.ts:95-127` handles `email`, `telegram`, and `push` channels with placeholder implementations that log and return without throwing. Since the job "succeeds," BullMQ marks it complete. Monitoring will show 100% notification success rate while no emails or messages are actually delivered. These should either throw (causing retry/failure) or be explicitly gated behind a feature flag that prevents jobs from being queued for unimplemented channels.

## Cron job sync with invalid provider ID

`cron/index.ts:18-30`:
```typescript
await syncQueue.add('scheduled-sync', {
  providerId: 'all',
  providerCode: 'all',
  syncType: 'price_only',
  triggeredBy: 'cron',
});
```

The sync worker (`sync.worker.ts:41-46`) does:
```typescript
const provider = await prisma.provider.findUnique({
  where: { id: providerId }, // 'all' is not a valid UUID
});
if (!provider || !provider.isActive) return; // silently exits
```

This means the scheduled sync never actually runs. The cron job should enumerate active providers and queue individual sync jobs for each.

## Missing env validation for critical secrets

`config/env.ts` validates only generic configuration (PORT, DATABASE_URL, JWT secrets). It does not validate:
- `PAYMENT_ENCRYPTION_KEY` (needed by PaymentGatewayFactory)
- Provider-specific API keys
- Webhook callback URLs
- Socket.io Redis adapter URL (when enabled)

A deployment missing `PAYMENT_ENCRYPTION_KEY` won't fail at startup — it'll fail on the first payment attempt.

## Rate limiter uses in-memory store

The `express-rate-limit` middleware (`rate-limiter.middleware.ts`) uses the default in-memory store. With multiple server instances, each instance has its own counter, effectively multiplying the rate limit by the number of instances. For a payment platform, this should use `rate-limit-redis` to share state.
